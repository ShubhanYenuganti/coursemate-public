## Context

The LLM synthesis pipeline (`api/llm.py`) currently returns replies as a 2-tuple `(content, summary)` parsed from JSON `{"reply": "...", "summary": "..."}`. Six distinct code paths in `api/chat.py` call `synthesize()` and unpack this tuple before inserting the assistant message. The `run_agent_openai` path handles streaming and has its own event emission loop.

Follow-up questions need to be generated by the same LLM call that produces the reply — no second call — and must flow all the way to the frontend as a distinct field so they can be rendered as interactive chips.

## Goals / Non-Goals

**Goals:**
- Every assistant reply carries 2-3 follow-up questions in a `follow_ups` JSON array
- Follow-ups are generated in the same LLM call as the reply (no extra API round-trip)
- Follow-ups are persisted to `chat_messages.follow_ups` (JSONB, nullable)
- Follow-ups are returned in the message API response payload
- Frontend renders them as chips; clicking loads text into the chat input box
- Works across all six code paths (send, stream, edit, regenerate, revert, restore)

**Non-Goals:**
- Auto-submitting follow-up chips (user must confirm/edit first)
- Generating follow-ups on clarification-request messages (future feature)
- Follow-ups on the revert/restore paths (those replay existing messages; follow-ups are preserved from the original stored row, not re-generated)

## Decisions

### 1. Generation inside `synthesize()`, not at the call site

Follow-up generation is added to the system prompt instruction and extracted from the LLM JSON output inside `synthesize()` (and mirrored in the `run_agent_openai` streaming path). This keeps all LLM orchestration in one place and avoids requiring every call site to know about the follow-up generation step.

**Alternative considered**: a post-synthesis call to a cheap model. Rejected — adds latency and an extra API call for something that costs essentially nothing when included in the main prompt.

### 2. `follow_ups` as a top-level JSON field, not embedded prose

The LLM output format expands from `{"reply": "...", "summary": "..."}` to `{"reply": "...", "summary": "...", "follow_ups": ["q1", "q2", "q3"]}`. This keeps the reply body clean (no trailing "Want to explore...?" prose) and gives the frontend a typed, structured list to render.

### 3. `synthesize()` return signature expands to 3-tuple

`synthesize()` currently returns `(content, summary)`. It expands to `(content, summary, follow_ups)` where `follow_ups` is a `list[str]` (possibly empty). All six unpack sites in `chat.py` must be updated together. The revert/restore paths pull follow-ups from the stored row rather than from a new `synthesize()` call.

### 4. JSONB column on `chat_messages`

`follow_ups JSONB NOT NULL DEFAULT '[]'` added to `chat_messages`. Nullable would also work, but an empty array default is cleaner to handle on the frontend without null-checks. Existing rows default to `[]`.

### 5. Chip click → populate input, no auto-submit

Clicking a chip calls the existing input setter (the `ChatTab` controlled input state or the forwarded ref) with the follow-up text, then focuses the input. This matches the UX spec: the user sees the pre-filled text and can edit, change model, or discard before sending.

## Risks / Trade-offs

- **LLM doesn't always comply with schema** → `_parse_synthesis_json` already has fallback logic; extend it to default `follow_ups` to `[]` when the field is absent or malformed. Never fail a reply because follow-ups couldn't be parsed.
- **Six unpack sites must all change atomically** → update all in one PR; a partial update would cause `ValueError` at runtime on the paths not yet updated.
- **Streaming path (`run_agent_openai`)** → follow-ups must be accumulated during streaming and emitted as a final `follow_ups` SSE event after the reply stream closes, then stored. The on_event callback already handles custom event types.
- **Revert/restore paths** → these replay a stored assistant message. They should read `follow_ups` from the original row and re-insert it, same as the `summary` field today.

## Migration Plan

1. Add `follow_ups JSONB NOT NULL DEFAULT '[]'` column to `chat_messages` (single ALTER TABLE, non-destructive, safe on live table).
2. Deploy backend changes (synthesize, chat.py paths, API response).
3. Deploy frontend chip rendering.
4. No rollback complexity — `follow_ups` defaults to `[]`; old clients ignore the field.
